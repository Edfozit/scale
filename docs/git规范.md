# Git 提交规范

> **本文档规定 AI 完成开发任务后的 Git 提交流程。AI 每次完成一个小功能或修复一个 bug 后，在用户确认没有问题后，应主动提醒用户按照本规范提交 Git。**

---

## 1. 提交时机

AI 在以下情况完成后，确认功能正常无误，应**主动提醒用户提交 Git**：

- 完成一个新功能（无论大小）
- 修复一个 Bug
- 完成一组相关的 UI 调整
- 完成一次重构

**原则**：小步提交，每次提交只做一件事，方便回滚和问题定位。

---

## 2. Commit Message 格式

采用 **类型 + 简短描述** 的格式，使用中文描述：

```
<类型>: <简短描述>
```

### 2.1 类型列表

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 添加饮食记录输入框` |
| `fix` | 修复 Bug | `fix: 修复图表页切换后数据不刷新` |
| `ui` | UI 样式调整 | `ui: 统一统计卡片字体大小` |
| `refactor` | 代码重构（不改变功能） | `refactor: 提取日历绘制逻辑到独立方法` |
| `docs` | 文档变更 | `docs: 更新项目架构文档` |
| `chore` | 构建/配置/杂项 | `chore: 升级 Room 依赖版本` |

### 2.2 描述规范

- 使用中文，简洁明了
- 描述「做了什么」，而不是「改了什么文件」
- 长度控制在 50 字以内

### 2.3 示例

```
feat: 添加运动记录输入功能
fix: 修复折线图日期标签被裁切问题
ui: 统一月度统计卡片间距为12dp
refactor: 重构WeightCalendarView月份切换逻辑
docs: 新增错误合集文档
chore: 更新AppDatabase版本号至3
```

---

## 3. 提交流程

```
1. AI 完成开发
       ↓
2. AI 自检（功能检查清单）
       ↓
3. 用户验证功能
       ↓
4. AI 提醒用户提交 Git
       ↓
5. 用户确认并执行提交
```

### 3.1 AI 提醒模板

当 AI 确认开发完成且用户验证通过后，使用以下格式提醒：

```
功能已完成并验证通过，建议提交 Git：

git add -A
git commit -m "<类型>: <描述>"

例如：
git add -A
git commit -m "feat: 添加运动记录输入功能"
```

---

## 4. 提交前检查

AI 在提醒提交前，应确认：

- [ ] 功能已实现且用户验证通过
- [ ] 没有引入新的编译错误
- [ ] 没有遗留的调试代码（如临时的 `Log.d`、注释掉的代码块）
- [ ] 相关文档已更新（如有必要）
- [ ] 新的踩坑经验已记录到 `docs/错误合集.md`（如有）

---

## 5. 分支策略

当前项目采用简单的分支策略：

| 分支 | 用途 |
|------|------|
| `main` / `master` | 稳定版本，功能完整时合并 |
| `dev` | 日常开发分支（推荐） |
| `feature/xxx` | 较大功能开发时创建独立分支 |
| `fix/xxx` | Bug 修复时创建独立分支（可选） |

> **建议**：小改动可以直接在 `dev` 分支提交；大功能建议创建 `feature/xxx` 分支，开发完成后合并回 `dev`。

---

## 6. 注意事项

- **不要提交构建产物**：`build/`、`.gradle/`、`local.properties` 等已在 `.gitignore` 中
- **不要 force push**：避免使用 `git push --force`
- **提交前检查 diff**：确认只包含本次改动的文件，没有误提交无关文件
