# AI 开发规范

> **本文档是 AI 辅助开发本项目的行为准则。AI 在每次任务开始前必须阅读本文件，严格遵守所有约束。**

---

## 1. 开发工作流

### 1.1 方案审核制

本项目采用 **AI 提方案 → 用户审核 → 确认后执行** 的开发模式：

1. **需求分析**：AI 先理解用户需求，必要时阅读相关代码和文档
2. **方案提出**：AI 给出具体的实现方案（涉及哪些文件、如何修改、为什么这样做）
3. **等待审核**：**禁止**在用户确认前直接动手修改代码
4. **执行开发**：用户确认方案后，AI 按计划实施
5. **验证测试**：完成后验证功能是否正常
6. **Git 提交**：确认无误后，按照 `git规范.md` 提交代码

### 1.2 必读文档

每次开发任务开始前，AI 必须阅读以下文档：

| 文档 | 作用 |
|------|------|
| `docs/项目架构.md` | 了解项目架构、技术栈、约束规范 |
| `docs/ai开发规范.md` | 了解开发行为规范（本文件） |
| `docs/错误合集.md` | 了解历史踩坑经验，避免重复犯错 |
| `docs/git规范.md` | 了解提交规范 |

## 2. 禁止事项（红线）

> **以下行为绝对禁止，违反将导致严重问题。**

### 2.1 架构层面

| # | 禁止行为 | 原因 |
|---|---------|------|
| B1 | 修改 `docs/项目架构.md` 中的架构设计 | 项目架构一旦确定不轻易修改，如需调整必须明确告知用户 |
| B2 | 引入新的架构模式（如 Navigation Component、Hilt/Dagger） | 与现有架构冲突 |
| B3 | 拆分子模块 | 项目为单模块结构 |
| B4 | 修改包名 `com.example.myapplication` | 影响全局，代价极高 |

### 2.2 技术层面

| # | 禁止行为 | 原因 |
|---|---------|------|
| T1 | 使用 `androidx.room.*` 包名 | 必须使用 `androidx.room3.*`（Room 3） |
| T2 | 使用 `findViewById` | 必须使用 View Binding |
| T3 | 使用 kapt 注解处理 | 必须使用 KSP |
| T4 | 在主线程执行数据库操作 | 必须通过 `viewModelScope.launch` 异步执行 |
| T5 | 修改实体后不更新 `AppDatabase` 版本号 | 会导致崩溃 |
| T6 | 体重单位使用公斤 | 统一使用「斤」 |

### 2.3 UI 层面

| # | 禁止行为 | 原因 |
|---|---------|------|
| U1 | 日历日期数字加粗 | 所有状态均使用普通字重 |
| U2 | 在布局 XML 中硬编码未注册的颜色值 | 必须先注册到 `colors.xml` |
| U3 | 只修改 `themes.xml` 不同步 `values-night/themes.xml` | 必须 light/dark 双主题同步 |
| U4 | 统计数据数值加粗 | 统计文本不加粗 |
| U5 | 日历视图随下方内容滚动 | 日历固定在顶部 |
| U6 | 发明新的字号/尺寸 | 必须使用项目架构.md 第 8.6 节 Token 速查表中的标准值 |

### 2.4 行为层面

| # | 禁止行为 | 原因 |
|---|---------|------|
| A1 | 未经用户确认直接修改代码 | 必须方案审核通过后才执行 |
| A2 | 删除用户没有要求删除的代码或文件 | 避免意外破坏 |
| A3 | 擅自重构不相关的代码 | 聚焦当前任务 |
| A4 | 在不确定时猜测文件内容 | 必须先读取确认 |

## 3. 开发准则

### 3.1 修改前必做

- **先读后改**：修改任何文件前，必须先完整读取该文件
- **先查后替**：执行文本替换前，先用 grep 确认目标文本确实存在
- **先备份后动**：大规模修改前，建议通过 git 创建分支或提交当前状态

### 3.2 修改时必做

- **最小化改动**：只修改与当前任务相关的代码，不动其他部分
- **上下文充足**：替换操作时，提供足够的前后文确保唯一定位
- **即时验证**：每个文件修改后检查是否有编译错误

### 3.3 修改后必做

- **功能检查清单**（逐项检查）：
  - [ ] Room 导入使用 `androidx.room3.*`
  - [ ] 修改实体后更新了 `AppDatabase` version 号
  - [ ] View 访问使用 View Binding（无 `findViewById`）
  - [ ] Fragment 数据刷新走 `onHiddenChanged` + ViewModel
  - [ ] 新增 UI 元素符合项目架构.md 第 8 章 UI 设计标准
  - [ ] 颜色值使用 `colors.xml` 已定义的资源
  - [ ] 主题修改同步覆盖 light 和 dark
  - [ ] 体重数值显示单位为「斤」
- **遇到新问题**：如果开发过程中遇到新的坑，主动记录到 `docs/错误合集.md`

## 4. Fragment 开发特别注意

本项目 Fragment 采用 **hide/show** 模式（非 Navigation Component），有以下强制要求：

| 规则 | 说明 |
|------|------|
| Fragment 实例缓存 | 通过 `lazy` 创建，避免每次切换重新创建 |
| 数据刷新时机 | 重写 `onHiddenChanged(hidden: Boolean)`，`hidden == false` 时刷新数据 |
| 不依赖 onResume | hide/show 模式下切换不会触发 onResume |
| 底部导航同步 | 新 Fragment 需在 `bottom_nav_menu.xml` 注册菜单项 + `MainActivity` 注册切换逻辑 |

## 5. 新增页面 / 功能检查清单

### 5.1 新增页面

1. 创建 Fragment + 布局 XML
2. 在 `bottom_nav_menu.xml` 添加菜单项
3. 在 `MainActivity` 注册切换逻辑（lazy 实例 + hide/show 管理）
4. 如需独立 ViewModel，创建对应的 ViewModel 类
5. 布局遵循项目架构.md 第 8 章 UI 设计标准

### 5.2 修改数据层

1. 修改 `WeightRecord` 实体
2. 更新 `WeightRecordDao` 接口（如有新查询需求）
3. **必须** 更新 `AppDatabase` 的 `version` 号
4. 确认 `fallbackToDestructiveMigration()` 已配置（schema 变更会清空数据）

2. ...
